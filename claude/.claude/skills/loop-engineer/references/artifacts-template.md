# Artifact Templates

Templates the `loop-engineer` skill emits. Fill the `<placeholders>`. Keep what's
relevant to the rung being scaffolded; don't dump every file at once.

Default output dir: `./loops/<slug>/` where `<slug>` is a kebab-case name for the
loop (e.g. `linkedin-article`, `python-speedup`).

---

## 1. LOOP-SPEC.md (the blueprint — written at step 3, before any code)

```markdown
# Loop Spec: <loop name>

_Generated by loop-engineer and shown inline as the loop is built. This records the
decisions baked into the construct in this folder._

## Task
<one paragraph: what the loop does and why>

## Verdict: is this a loop?
<YES, building it / NO — one-shot, use /goal / NO — success undefinable>. <reasoning:
name the success criterion and the horizon. Only NO halts the build.>

## Maturity rung
Current: <0–4, label>. <If 0–1: validation gate is baked into run #1. If 2+: goes

## The four phases
- **Trigger:** <schedule/cron/webhook/manual + cadence>
- **Execution:** <skill name; new or existing; what it does>
- **Verification:** Tier <1–5> — <exactly how success is measured>
  - <if tier 4: which judge model, why not the executor>
  - <if tier 5: what the human approves and when>

  - Measurement lag: <none | second scraper loop needed, describe>

## Stop rule
<goal-hit | no-progress plateau | hard cap N / token budget — pick at least one hard stop>

## Open risks
<anything fuzzy, any proxy-metric danger, any unproven assumption>

## Files in this construct
<list of everything generated into loops/<slug>/>
```

## 2. The validation gate (rung 0–1 — task not yet proven)

You still build the full loop. You just make its **first run** prove the task before
the loop trusts its own scores. This is baked into the execution skill (section 3) via
a `validated` flag in state — not a separate "go do homework first" file.

Behavior to encode in the execution skill:

```markdown
## Validation gate (first run only)
Read `validated` from state.
- If `validated` is false → run in VALIDATION MODE: produce exactly ONE output, then
  STOP and ask the human: "Is this acceptable as a baseline? (yes/no)". On "yes", set
  `validated: true` and `baseline` to this run in state. On "no", incorporate the
  human's note and try once more — do NOT enter the optimization loop.
- If `validated` is true → run normally (optimize mode), trusting the verification
  scores and prior state.

Why: looping an unproven task just automates failure. Run #1 proves it's possible;
everything after improves on a proven baseline.
```

Optionally also write a short `MANUAL-RUN.md` the human can follow by hand if they
prefer to validate outside the loop first — but the gate above means they don't have
to. The loop is delivered either way.

## 3. Execution skill (rung 1 → 2)

Write to `loops/<slug>/skill/SKILL.md`. This is a normal Claude Code skill.

```markdown
---
name: <slug>
description: >-
  <what the task produces and when to run it. If this skill will be fired by a
  trigger inside a loop, say so.>
---

# <Loop name> — execution

## 0. Validation gate (only if the task was unproven when built)
Read `validated` from `../state.json`. If false, run in VALIDATION MODE: produce one
output, stop, and ask the human to confirm it as a baseline before the loop optimizes.
On confirm, set `validated: true`. (Omit this section entirely for rung 2+ tasks that
were already proven.)

## 1. Read state first (self-improvement)
Read the loop state at `../state.json`. Review prior runs: what was tried, the scores,
what worked and what didn't. Bring that into this run — never repeat a logged dud.

## 2. Do the work
<imperative steps to perform the task a specific way for a specific output>

## 3. Verify + record
Run the verification (see `../verify.md`), then append this run's result to state
(see `../state.schema.json`): approach, score, what worked, what failed.
```

## 4. trigger.md (rung 2 → 3)

```markdown
# Trigger: <loop name>

Cadence: <e.g. daily 09:00>.

> loop-engineer offers to wire ONE of these for you at the end (Step 4 — Activate).
> Nothing is scheduled until you say yes and confirm the cadence.

## Option A — Claude Code routine / scheduled task
Create a routine/automation that runs the `<slug>` skill on schedule. In the
routine instructions, simply: "Run the <slug> skill." Set the schedule to <cadence>.
Simplest if you live in Claude Code.

## Option B — OS scheduler (portable, survives Claude Code being closed)
- macOS/Linux cron: `0 9 * * *  cd <repo> && claude -p "run the <slug> skill"`
- Windows Task Scheduler: a task on the same cadence running
  `claude -p "run the <slug> skill"` with the working dir set to <repo>.

## Option C — Webhook / event
<only if event-driven: describe the event and the handler that invokes the skill>

## Option D — Manual
Leave nothing scheduled; run `<slug>` yourself when you want. Fine while you're still
tuning the loop.

Pick the simplest that fits. Automating the trigger is independent of full
loop-ification — you can wire it before or after adding state/verification.
```

## 5. state.schema.json + verify.md + stop.md (rung 3 → 4)

### state.schema.json
```json
{
  "loop": "<slug>",
  "validated": false,
  "baseline": null,
  "runs": [
    {
      "run_id": "<iso-timestamp or seq>",
      "timestamp": "<iso8601>",
      "inputs": "<what this run was given>",
      "artifact_ref": "<path/url to what it produced>",
      "approach": "<what was tried this run — hooks, params, diffs, angle>",
      "score": "<the metric value, or judge verdict, or 'pending' if lagged>",
      "worked": "<what helped>",
      "failed": "<what didn't>"
    }
  ]
}
```

### verify.md
```markdown
# Verification: <loop name> — Tier <N>

## How success is measured
<the concrete check>

### Tier 1–3 (objective)
Run <command/script> → read <metric>. Pass if <condition>. Record `score` in state.

### Tier 4 (judge — different model than executor)
Send the artifact to <judge model, e.g. Codex> with this prompt:
> "<judge prompt — score the artifact on <criteria> from 1–10, list what works and
> what doesn't. Be adversarial; do not flatter.>"
Record the verdict + reasons in state. NEVER let the executor judge its own output.

### Tier 5 (human gate)
Stop and present the artifact to the user. Do not mark success autonomously. Record
the human's decision + notes in state.

## Measurement lag
<if the score arrives later: describe the second loop that scrapes outcomes and
backfills `score` for past runs by `run_id`>
```

### stop.md
```markdown
# Stop rule: <loop name>

Stop looping when ANY of:
- Goal hit + verified: <condition>
- No progress: <metric> hasn't improved over the last <N> runs.
- Hard cap: <N iterations> or <token/$ budget> reached.

Always keep at least one hard cap — AI isn't free.
```
